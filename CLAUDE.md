# CLAUDE.md — itbisa-shop-report-bot

> **Source of truth for the Claude surfaces** — read by **Claude Code** and pasted into the **Claude Chat** project instructions (Claude has no tight char cap, so it gets the full detail). `AGENTS.md` (ChatGPT Codex) points here; `CHATGPT_CHAT.md` is the ≤8000-char condensed copy for **ChatGPT Chat** only.

Standalone, **offline** Python tool that turns ITBisa sales/stock Excel exports into an analysis workbook (sales performance, pricing, supplier, and reorder analysis + A/B price-test reports). No API calls, no network, no tokens, no GitHub Actions — runs locally and is idempotent.

> Unlike the sibling ITBisa repos, this one is **not** a GitHub-Actions bot. It does not call Shopee/TikTok Shop APIs, has no `bot-state` branch, no workflows, no Telegram, and no secrets.

## Stack & files (flat layout, no `src/`)
- Python 3.13. Deps: `pandas` (pinned `>=2.0,<3.0`), `openpyxl` (`requirements.txt`). The pin caps below pandas 3.0 because the co-located `laporan/` generator isn't ported to 3.0 yet (its new `str` dtype + Copy-on-Write default need separate work).
- `main.py` — CLI entry point / orchestration.
- `config.py` — all constants: globs, sheet/column names, thresholds, colors, supplier keywords, reorder + A/B params.
- `data_loader.py` — multi-file glob loading, SKU normalization, current-workbook loaders (arrived beli / jual / hilang / pindah).
- `analysis.py` — HPP weighted-average, profit, per-SKU aggregation, supplier classification, reorder metrics, `build_stock_ledger`.
- `tables.py` — table builders (diminati, profit, rugi, kandidat, supplier, reorder).
- `excel_writer.py` — Excel output, incl. the per-gudang stock-reconciliation sheet.
- `ab_testing.py` — A/B price-test analyzer + template creation.
- `restock_pricing.py` — restock price evaluator (predict landed HPP, judge supplier price, recommend per-marketplace selling price) + template.
- `cashflow.py` — cash-flow restock plan: from the reorder metrics, project each order's timing/qty/cost (all cycles in the horizon) and bucket the Rupiah by month & supplier.
- `deadstock_analysis.py` — dead-stock / capital-release: Rupiah frozen in Overstock + Slow/Dead SKUs and how to free it.
- `trend_analysis.py` — sales trend & seasonality: cross-year omzet/profit trend, YoY growth, per-month seasonal index.
- `stock_opname.py` — stock-opname reconciliation: compare a physical count (`data/stock_opname.xlsx`) vs the bot ledger `sisa_stok` and emit `BisaHilang` adjustment rows (Banyak/Nilai Hilang/Ketemu) to paste into the Google Sheets BisaHilang tab.
- `data/` — input Excel (gitignored). `output/` — generated reports.
- `laporan/` — the **co-located Laporan generator** (an importable **`laporan` package**: `invoice/jual/remit/bonus/fee/final/process/rekonsiliasi/keywordchecker/utility/` subpackages with `__init__.py`, its own `main.py` orchestration/CLI + `__main__.py`, and sample `data/`), merged in with full history. Turns raw marketplace exports (`Transaksi`/`Saldo`/`Fee`) into `Laporan` workbooks; it builds `Invoice`/`Jual`/`Remit`/`Bonus`/`Final` internally, then collapses each workbook to its **deliverable sheets**: `Jual <MP>` + `Remit <MP>` (the combined `Final` reconciliation, promoted/renamed) + `Bonus <MP>` where present. Run it via `python main.py --laporan` (or standalone with `python -m laporan`) — it's called **in-process** (no subprocess); every intra-package import is the absolute `laporan.*` form, so the generator's modules **never collide** with this tool's flat root modules. The readers tolerate the marketplaces' evolving exports (TikTok `Fee` English **or** Indonesia-localized columns; TikTok order export with a broken `.xlsx` dimension; Shopee `Fee` `Adjustment` floating header). Its `Jual` sheet feeds this bot's `Jual` ledger through a **manual Google Sheets step** (copy Laporan → Jual Sheets → export `Jual*.xlsx` into `data/`), so the **generate and analyze stages are run separately, not auto-chained**.

## CLI (`python main.py`)
- (no flag) → **runs the full suite (same as `--all`)**: all years sales + trend + reorder + cash-flow + dead-stock + ab-test + restock-check.
- `--sales` → all years (one file per year); `--sales YEAR` → specific year.
- `--trend` → `Analisa_Tren_Musiman.xlsx` (cross-year omzet/profit trend, YoY growth, seasonal index; zero-config).
- `--reorder` → standalone `Analisa_Reorder.xlsx`.
- `--cashflow` → `Analisa_Cashflow_Restock.xlsx` (restock budget calendar; zero-config, built from the reorder metrics).
- `--deadstock` → `Analisa_Modal_Beku.xlsx` (capital frozen in slow/dead/overstock + how to free it; zero-config).
- `--ab-test` → `Analisa_AB_Test.xlsx` (auto-creates `data/ab_tests.xlsx` template if missing).
- `--restock-check` → `Analisa_Restock_Check.xlsx` (auto-creates `data/restock_check.xlsx` template if missing).
- `--stock-opname` → `BisaHilang_Rekonsiliasi.xlsx` (auto-creates `data/stock_opname.xlsx` template if missing). Reconciles a physical count vs the ledger `sisa_stok`: per SKU `selisih = stok_buku − stok_fisik` → **Banyak Hilang** (>0) or **Banyak Ketemu** (<0), valued at `qty × HPP` (`STOCK_OPNAME_VALUE_BASIS`, default `hpp_wa`; FIFO not supported). Gudang = the template's `Lokasi Gudang` else the SKU's dominant ledger gudang (else the busiest warehouse). Output sheet `BisaHilang` (discrepancy rows only) is pasteable into the Google Sheets BisaHilang tab; standalone (not part of `--all`). See `stock_opname.py`.
- `--all` → all years + trend + reorder + cash-flow + dead-stock + ab-test + restock-check (7 steps). **Loads the workbooks once (`_load_shared`) and computes reorder once**, sharing them across every step (the standalone flags still load on their own). Sales, trend, cash-flow, and dead-stock need no template, so they always run; the ab-test and restock-check steps run **only when their template exists with rows** — otherwise they're skipped silently (no template is auto-created, so the full run never halts). Use the dedicated `--ab-test` / `--restock-check` flag to bootstrap a template.
- `--data-dir` / `--output-dir` override the defaults.
- `--laporan` → run the **Laporan generator** in `laporan/` (raw exports → `Laporan` workbooks, delivered as `Jual` + `Remit`(=`Final`) + `Bonus`). No arg = every marketplace; scope it: `--laporan shopee tiktok`. Reads `laporan/data`, writes `laporan/reports`. Runs **in-process** (`run_laporan` calls `laporan.main.main`); a terminal action — does not also run the analysis suite (the Jual Sheets refresh is a manual step in between).

## Inputs (`data/`, matched by glob)
- `STOK_GLOB = "*Stok*.xlsx"` (purchases) and `JUAL_GLOB = "*Jual*.xlsx"` (sales). Filenames must contain `Stok` / `Jual`.
- Stok files need sheet `Stok`; the latest file also needs `Hilang` and `PindahBarang` for stock reconciliation.
- Jual files need at least `JualShopee`; other `Jual*` sheets load if present.
- Stok's **`Toko`** column holds the **standardized** supplier/forwarder (e.g. `Ocistok/Martkita`, `AliExpress`, `Jasa Impor Tiongkok`, `Osell`, `Shopee`, `Tokopedia`, local distributors); the payment account + invoice/resi live in **`Keterangan Pembelian`** (not read by the tool); **`Luar Negeri?`** is the authoritative overseas flag (it splits e.g. `Shopee` into local vs cross-border import). The loader auto-renames the pre-standardization header `Toko[spasi]Akun Pemesan` → `Toko` (`COL_STOK_TOKO_LEGACY`).
- **Naming / backward-compat (de-branding migration)**: the `Bisa` prefix was dropped from every artifact name (`BisaJual`→`Jual`, `BisaStok`→`Stok`, sheets `BisaJualShopee`→`JualShopee`, the `Laporan` workbook, the generator's `Invoice`/`Remit`/`Bonus` sheets, …). The loaders still accept the **legacy `Bisa*` names** transparently: the globs match both (`*Stok*` / `*Jual*`) and **`resolve_sheet()`** (in `data_loader.py`) falls back to the `Bisa`-prefixed sheet — so existing data files + Google Sheets keep working with no change. To fully retire the legacy names, rename your Google Sheets tabs and re-export. The `ITBisa` brand itself (company name, `ITBisa.com` file prefix) is unchanged — only the `Bisa*` artifact prefixes were removed.

## Output: `output/Analisa_Penjualan_ITBisa_<year>.xlsx`
**9 sheets (pure sales history)**: `00_Summary`, `01_Paling_Diminati`, `02_Profit_Tertinggi`, `03_Barang_Rugi`, `04_Margin_Borderline`, `05_Kandidat_Naik_Harga`, `06_Per_Platform`, `07_Data_Lengkap_per_SKU`, `08_Supplier_Analysis`. **Reorder + per-gudang stock are a current-day snapshot (not year-specific), so they are NOT written into the yearly file** — they live only in `Analisa_Reorder.xlsx` (`--reorder`: `00_Reorder_Summary`, `01_Reorder_Action`, `02_Reorder_Data_Lengkap`, `03_Rekap_Stok_per_Gudang`). `ledger_df` is still passed to the yearly writer for the `00_Summary` OVERSOLD data-quality note.

## Core logic (do not regress — README has the full spec)
- **Sisa stok = current-workbook ledger**, recomputed to match the Google Sheets `RekapBarang`: per (SKU, gudang) `arrived beli − nonvoid jual + ketemu − hilang ± pindah`. "Current workbook" = the latest `*Stok*`/`*Jual*` file by filename sort. Migrasi rows are the opening balance (kept here). Negative gudang balances floor to 0 and shift the deficit; truly OVERSOLD SKUs stay negative and are flagged to console.
- **HPP = weighted average with Ocistok-priority**: if a SKU has Ocistok/Martkita (China-direct) purchases, average those only; else average all. Combined across all stok files. For HPP/total-beli, Migrasi rows are dropped when real purchases exist (avoid double-count).
- **Pricing-decision basis** (markup %, Kandidat Naik Harga, Margin Borderline, Barang Rugi recommendations): `markup_pct = (harga_sekarang − hpp_pricing) / hpp_pricing`, where `hpp_pricing` (the **"HPP/buah"** column — renamed from "HPP Dasar Harga") is a **tiered** replacement cost and `harga_sekarang` = lowest non-CoD unit price (Omzet/Qty) on the SKU's most recent selling day (else `harga_jual_avg`). **Profit/margin still use `hpp_wa` + `harga_jual_avg`** (realized P&L). `hpp_pricing` tiers: **(1)** SKU with any overseas lot (`Luar Negeri? = 1`) → latest overseas lot price by Tanggal Bayar (consistent import price, no averaging); **(2)** purely-domestic SKU → qty-weighted average of its domestic lots in the most recent restock window that has data — **3 months, else 6, else 12** (`PRICING_HPP_WINDOWS_MONTHS`, anchored at today, Migrasi/undated lots excluded), so the basis tracks the **current domestic restock cost** not old lots; **(3)** fallback → all-time `hpp_wa` (no dated domestic lot in any window). The per-SKU tier is shown in Sheet 07 **"Sumber HPP Harga"** (`LN-terakhir` / `WA 3bln` / `WA 6bln` / `WA 12bln` / `WA`); the realized P&L cost is **"HPP/buah (P&L)"**. `hpp_pricing` is also the replacement cost in `--cashflow`. See `_latest_ln_price` / `_domestic_windowed_hpp` / `compute_harga_sekarang` in `analysis.py`.
- **Recent-price-increase guard (Kandidat Naik Harga)**: a SKU whose `harga_sekarang` is a freshly-raised, under-validated price is **held** — its `Harga +10/15/20%` and `Proyeksi Profit` cells are blanked and the Saran is replaced with `⏳ Harga baru naik … — kumpulkan data dulu` — because its `Qty Terjual`/`Profit` were earned at the OLD price, so extrapolating them at the new price is invalid. Change date comes from `ab_tests.xlsx` when logged (authoritative), else auto two-window step-detection; flags only when `harga_sekarang` is ≥`PRICE_CHANGE_MIN_STEP` above the pre-change price AND post-change qty is `< PRICE_CHANGE_VALIDATION_MIN_SHARE` of the year's qty. A long-stable price (e.g. Rp999 for months) is **not** flagged even if far above the lifetime average. See `compute_price_change_status` in `analysis.py`; `PRICE_CHANGE_*` tunables in `config.py`.
- **SKU normalization**: `UPPER().strip()` on load everywhere (matches case-insensitive `SUMIF`).
- **No dedup**: the old `drop_duplicates` was removed (it discarded genuine duplicate purchase lots); only drop-Migrasi remains.
- **Reorder**: velocity = avg monthly qty over the trailing N-month window, computed as **total ÷ N months** (not ÷ the calendar buckets the date-window straddles, which is N+1 and would dilute the rate by ~1/(N+1)); 6mo window, fallback 12→24mo. Volatility CV → safety multiplier; **lead time is per-SHOP, not per-SKU** (the forwarder ships at its own pace — AliExpress ≈1mo vs the Ocistok/Martkita sea-freight forwarder ≈2.5mo): each import shop's lead = `LEAD_TIME_PERCENTILE` (p75) of its observed `Tanggal Bayar`→`Tanggal Sampai` (non-Migrasi), thin shops (<`LEAD_TIME_MIN_LOTS`) → global-import p75. **Ocistok = Martkita = 1688 are one forwarder** (`OCISTOK_KEYWORDS`/`IMPORT_SHOP_KEYWORDS`). A SKU takes the **slowest shop supplying ≥`LEAD_SHOP_MIN_SHARE` of its qty** (plan for the slow import, not an occasional fast local top-up); import status comes from the reliable `Luar Negeri?`/China-keyword qty share and the forwarder from the **standardized `Toko`** value, and import SKUs are floored at the global-import lead. Local-sourced → `LEAD_TIME_MARKET_MONTHS`. Then ROP, suggested order; action buckets STOCKOUT/URGENT/Now/Soon/Overstock + Slow/Dead. Tunables in `config.py`. See `compute_lead_time_months`.
- **On-order / in-transit (already-purchased, not-yet-arrived)**: the stock ledger only counts **arrived** lots (`load_current_stok_arrived` keeps rows with `Tanggal Sampai` filled), so a SKU you've already re-bought but whose shipment hasn't landed showed low `sisa_stok` and got re-flagged URGENT. Fix: `_compute_on_order` sums per-SKU **in-transit qty** = non-Migrasi lots that are paid (`Tanggal Bayar` filled) but not arrived (`Tanggal Sampai` blank), paid within `ONORDER_MAX_AGE_MONTHS` (default 6, so old blank-Sampai data gaps don't false-flag). The reorder decision uses the **inventory position = `sisa_stok` + `qty_on_order`** (physical status is still classified on on-hand `sisa`, so a 0-on-hand SKU still reads as physically stocked out): `qty_order_suggest` nets out the incoming (only asks for the remainder), and a SKU that would be a buy-now bucket (STOCKOUT/URGENT/Now/Soon) whose incoming covers `rop_final` is relabelled **`⏳ Sudah Dipesan`** (`STATUS_SUDAH_DIPESAN`) with an **`est_arrival` = last order date + lead** — don't re-buy, wait for arrival. New reorder columns `qty_on_order` / `inventory_position` / `last_order_date` / `est_arrival`; a dedicated `⏳ Sudah Dipesan` section + a `Qty Dipesan` column in `01_Reorder_Action`, and the columns in `02_Reorder_Data_Lengkap`. `--cashflow` is **not** yet position-aware (still plans off `sisa`). See `_compute_on_order` in `analysis.py`.
- **Summary (00) surfacing**: the sheet flags a **partial/current year** (data not yet full-year, don't compare straight to complete years) and a **data-quality** block (OVERSOLD SKUs from the ledger + SKUs sold without HPP) so they're visible in the workbook, not just console.
- **Restock price check (`--restock-check`)**: input `data/restock_check.xlsx` (SKU, Toko, `Harga RMB` and/or `HPP IDR`, `Kompetitor Min`/`Max`). Predicts **landed** HPP from the raw RMB price via a factor **calibrated on the Ocistok/Martkita channel** (median of realized `HPP per Buah (Rp) ÷ (x RMB)` from the `Keterangan` note — per-SKU when it has ≥`RESTOCK_RMB_MIN_LOTS` lots, else the global median ≈ Rp`RMB_TO_IDR_FALLBACK`/RMB). This factor **already embeds the forwarder's margin + shipping + import** (it runs ~25% above the raw RMB→IDR spot `RMB_SPOT_FX_IDR`, because it's the final cost actually paid). A given `HPP IDR` overrides. Cost verdict = landed HPP vs the SKU's `hpp_wa` (±`RESTOCK_COST_TOL`). Per-marketplace selling price = `HPP × (1 + RESTOCK_TARGET_NET_MARKUP) / (1 − fee)` so net profit ≥ target AFTER the fee; **fees derived per platform from `Jual`** (`|admin|/omzet`, fallback `PLATFORM_FEE_FALLBACK`). Decision vs the competitor range: 🟢 restock & sell (hits target within market), 🟡 thin (profitable only below target), 🔴 don't sell (loss even at the competitor's max). See `restock_pricing.py`.
- **Cash-flow restock plan (`--cashflow`)**: turns the reorder metrics into a purchasing-budget calendar. For each SKU with demand, the **next** order is timed by when projected stock crosses `rop_final` (`months_to_order = max(0, (sisa − rop_final) / velocity)`; STOCKOUT/below-ROP → due now); only orders falling within `CASHFLOW_HORIZON_MONTHS` (default 6) are budgeted. Order qty reuses the reorder formula (`target_qty_post_reorder − stock_at_order + lead_demand`, ceil); **cost = qty × replacement HPP** = `hpp_pricing` (latest overseas lot, the cost you'd actually pay to restock), fallback `hpp_wa`. Supplier = the SKU's dominant standardized `Toko` by non-Migrasi qty. Output `Analisa_Cashflow_Restock.xlsx`: `00_Ringkasan` (total + due-this-month + per-month), `01_Kalender_per_Bulan` (supplier × month Rp matrix), `02_Detail_per_SKU` (one row per order, with `Order ke-`). Zero-config (no template), so it always runs in `--all`. Plans **all** reorder cycles in the horizon via an inventory-position simulation (order up to `target + lead_demand` whenever the position crosses `rop_final`, then deplete at `velocity`), so a fast mover gets several orders, capped by `CASHFLOW_MAX_CYCLES`. See `build_restock_plan` in `cashflow.py`.
- **Dead-stock / capital release (`--deadstock`)**: quantifies capital frozen in the reorder buckets `🔵 Overstock` and `💤 Slow/Dead`. Per SKU: `held = sisa_stok × hpp_wa` (capital tied up at cost) and `freeable = max(0, sisa_stok − target_qty_post_reorder) × hpp_wa` (the actionable excess above the reorder target). Recommendation: 🧹 Likuidasi (velocity ≤ 0 or no sale in `DEADSTOCK_DEAD_DAYS`), 🏷️ Markdown (slow turnover), ⛔ Stop reorder (healthy demand but far above target). Output `Analisa_Modal_Beku.xlsx`: `00_Ringkasan`, `01_Modal_Beku_per_SKU`, `02_Per_Supplier`. Zero-config (built from the reorder metrics). See `analyze_deadstock` in `deadstock_analysis.py`.
- **Sales trend & seasonality (`--trend`)**: a cross-year view the isolated per-year files can't give. **Yearly** omzet/qty/profit/orders + YoY omzet growth (current year flagged partial); **monthly** full omzet/profit time series (gaps filled 0); **seasonality** = a per-calendar-month index where each `(year, month)` omzet is divided by that year's own monthly average (de-trends for year-over-year growth) and averaged across the **complete** years only (the partial current year is excluded so it can't bias the pattern) — index > 1 = consistently strong, classed Puncak/Normal/Sepi via `TREND_PEAK_INDEX`/`TREND_LOW_INDEX`, thin months (< `TREND_SEASONAL_MIN_YEARS`) marked. Headline also shows YTD-vs-same-period-last-year. `profit = omzet + admin − hpp_wa × qty` (no-HPP SKUs count in omzet, not profit). Output `Analisa_Tren_Musiman.xlsx`: `00_Ringkasan`, `01_Tren_Tahunan`, `02_Tren_Bulanan`, `03_Musiman`. Zero-config. See `analyze_trend` in `trend_analysis.py`.
- **Supplier classification**: China = `Luar Negeri? = 1` or the standardized `Toko` matches `CHINA_KEYWORDS` (Ocistok/Martkita, AliExpress, Jasa Impor, 1688, Alibaba, Osell); Market = `MARKET_KEYWORDS` (Shopee/Tokopedia/Bukalapak/Blibli/Tiktok — matched exactly against the clean `Toko`); else Other. The keyword lists are tuned to the standardized `Toko` values (no trailing-space hacks).
- **Deterministic table ordering**: tables that sort on a tie-prone key — `08_Supplier_Analysis` (the comparison list + the single-source `head(15)` cut) and the reorder buckets — sort with a unique **`SKU` tiebreaker**, because pandas' default *unstable* sort otherwise ordered ties (and the `head` cut) differently across Python/pandas versions. See `build_supplier_analysis` / `build_reorder_tables` in `tables.py`.
