# CLAUDE.md ‚ÄĒ itbisa-shop-report-bot

> **Source of truth for the Claude surfaces** ‚ÄĒ read by **Claude Code** and pasted into the **Claude Chat** project instructions (Claude has no tight char cap, so it gets the full detail). `AGENTS.md` (ChatGPT Codex) points here; `CHATGPT_CHAT.md` is the ‚Č§8000-char condensed copy for **ChatGPT Chat** only.

Standalone, **offline** Python tool that turns ITBisa sales/stock Excel exports into an analysis workbook (sales performance, pricing, supplier, and reorder analysis + A/B price-test reports). No API calls, no network, no tokens, no GitHub Actions ‚ÄĒ runs locally and is idempotent.

> Unlike the sibling ITBisa repos, this one is **not** a GitHub-Actions bot. It does not call Shopee/TikTok Shop APIs, has no `bot-state` branch, no workflows, no Telegram, and no secrets.

## Stack & files (flat layout, no `src/`)
- Python 3.10+. Deps: `pandas`, `openpyxl` (`requirements.txt`).
- `main.py` ‚ÄĒ CLI entry point / orchestration.
- `config.py` ‚ÄĒ all constants: globs, sheet/column names, thresholds, colors, supplier keywords, reorder + A/B params.
- `data_loader.py` ‚ÄĒ multi-file glob loading, SKU normalization, current-workbook loaders (arrived beli / jual / hilang / pindah).
- `analysis.py` ‚ÄĒ HPP weighted-average, profit, per-SKU aggregation, supplier classification, reorder metrics, `build_stock_ledger`.
- `tables.py` ‚ÄĒ table builders (diminati, profit, rugi, kandidat, supplier, reorder).
- `excel_writer.py` ‚ÄĒ Excel output, incl. the per-gudang stock-reconciliation sheet.
- `ab_testing.py` ‚ÄĒ A/B price-test analyzer + template creation.
- `restock_pricing.py` ‚ÄĒ restock price evaluator (predict landed HPP, judge supplier price, recommend per-marketplace selling price) + template.
- `cashflow.py` ‚ÄĒ cash-flow restock plan: from the reorder metrics, project each SKU's next-order timing/qty/cost and bucket the Rupiah by month & supplier.
- `channel_analysis.py` ‚ÄĒ per-SKU channel optimizer: realized net margin/unit per marketplace, recommend the best channel.
- `basket_analysis.py` ‚ÄĒ bundle / cross-sell market basket: SKU pairs frequently bought in one `Invoice`.
- `data/` ‚ÄĒ input Excel (gitignored). `output/` ‚ÄĒ generated reports.

## CLI (`python main.py`)
- (no flag) ‚Üí **runs the full suite (same as `--all`)**: all years sales + reorder + cash-flow + channel + bundle + ab-test + restock-check.
- `--sales` ‚Üí all years (one file per year); `--sales YEAR` ‚Üí specific year.
- `--reorder` ‚Üí standalone `Analisa_Reorder.xlsx`.
- `--cashflow` ‚Üí `Analisa_Cashflow_Restock.xlsx` (restock budget calendar; zero-config, built from the reorder metrics).
- `--channel` ‚Üí `Analisa_Channel_per_SKU.xlsx` (per-SKU best marketplace by net margin; zero-config).
- `--bundle` ‚Üí `Analisa_Bundle_CrossSell.xlsx` (SKU pairs bought together; zero-config).
- `--ab-test` ‚Üí `Analisa_AB_Test.xlsx` (auto-creates `data/ab_tests.xlsx` template if missing).
- `--restock-check` ‚Üí `Analisa_Restock_Check.xlsx` (auto-creates `data/restock_check.xlsx` template if missing).
- `--all` ‚Üí all years + reorder + cash-flow + channel + bundle + ab-test + restock-check (7 steps). **Loads the workbooks once (`_load_shared`) and computes reorder once**, sharing them across every step (the standalone flags still load on their own). Cash-flow, channel, and bundle need no template, so they always run; the ab-test and restock-check steps run **only when their template exists with rows** ‚ÄĒ otherwise they're skipped silently (no template is auto-created, so the full run never halts). Use the dedicated `--ab-test` / `--restock-check` flag to bootstrap a template.
- `--data-dir` / `--output-dir` override the defaults.

## Inputs (`data/`, matched by glob)
- `STOK_GLOB = "*BisaStok*.xlsx"` (purchases) and `JUAL_GLOB = "*BisaJual*.xlsx"` (sales). Filenames must contain `BisaStok` / `BisaJual`.
- Stok files need sheet `BisaStok`; the latest file also needs `BisaHilang` and `BisaPindahBarang` for stock reconciliation.
- Jual files need at least `BisaJualShopee`; other `BisaJual*` sheets load if present.
- BisaStok's **`Toko`** column holds the **standardized** supplier/forwarder (e.g. `Ocistok/Martkita`, `AliExpress`, `Jasa Impor Tiongkok`, `Osell`, `Shopee`, `Tokopedia`, local distributors); the payment account + invoice/resi live in **`Keterangan Pembelian`** (not read by the tool); **`Luar Negeri?`** is the authoritative overseas flag (it splits e.g. `Shopee` into local vs cross-border import). The loader auto-renames the pre-standardization header `Toko[spasi]Akun Pemesan` ‚Üí `Toko` (`COL_STOK_TOKO_LEGACY`).

## Output: `output/Analisa_Penjualan_ITBisa_<year>.xlsx`
12 sheets: `00_Summary`, `01_Paling_Diminati`, `02_Profit_Tertinggi`, `03_Barang_Rugi`, `04_Margin_Borderline`, `05_Kandidat_Naik_Harga`, `06_Per_Platform`, `07_Data_Lengkap_per_SKU`, `08_Supplier_Analysis`, `09_Reorder_Analysis`, `10_Reorder_Data_Lengkap`, `11_Rekap_Stok_per_Gudang`. The reorder report (`--reorder`) emits its own workbook with a `03_Rekap_Stok_per_Gudang` sheet.

## Core logic (do not regress ‚ÄĒ README has the full spec)
- **Sisa stok = current-workbook ledger**, recomputed to match the Google Sheets `BisaRekapBarang`: per (SKU, gudang) `arrived beli ‚ąí nonvoid jual + ketemu ‚ąí hilang ¬Ī pindah`. "Current workbook" = the latest `*BisaStok*`/`*BisaJual*` file by filename sort. Migrasi rows are the opening balance (kept here). Negative gudang balances floor to 0 and shift the deficit; truly OVERSOLD SKUs stay negative and are flagged to console.
- **HPP = weighted average with Ocistok-priority**: if a SKU has Ocistok/Martkita (China-direct) purchases, average those only; else average all. Combined across all stok files. For HPP/total-beli, Migrasi rows are dropped when real purchases exist (avoid double-count).
- **Pricing-decision basis** (markup %, Kandidat Naik Harga, Margin Borderline, Barang Rugi recommendations): `markup_pct = (harga_sekarang ‚ąí hpp_pricing) / hpp_pricing`, where `hpp_pricing` = latest overseas lot price (`Luar Negeri? = 1`, by Tanggal Bayar; else `hpp_wa`) and `harga_sekarang` = lowest non-CoD unit price (Omzet/Qty) on the SKU's most recent selling day (else `harga_jual_avg`). **Profit/margin still use `hpp_wa` + `harga_jual_avg`** (realized P&L). See `_latest_ln_price` / `compute_harga_sekarang` in `analysis.py`.
- **Recent-price-increase guard (Kandidat Naik Harga)**: a SKU whose `harga_sekarang` is a freshly-raised, under-validated price is **held** ‚ÄĒ its `Harga +10/15/20%` and `Proyeksi Profit` cells are blanked and the Saran is replaced with `‚Ź≥ Harga baru naik ‚Ä¶ ‚ÄĒ kumpulkan data dulu` ‚ÄĒ because its `Qty Terjual`/`Profit` were earned at the OLD price, so extrapolating them at the new price is invalid. Change date comes from `ab_tests.xlsx` when logged (authoritative), else auto two-window step-detection; flags only when `harga_sekarang` is ‚Č•`PRICE_CHANGE_MIN_STEP` above the pre-change price AND post-change qty is `< PRICE_CHANGE_VALIDATION_MIN_SHARE` of the year's qty. A long-stable price (e.g. Rp999 for months) is **not** flagged even if far above the lifetime average. See `compute_price_change_status` in `analysis.py`; `PRICE_CHANGE_*` tunables in `config.py`.
- **SKU normalization**: `UPPER().strip()` on load everywhere (matches case-insensitive `SUMIF`).
- **No dedup**: the old `drop_duplicates` was removed (it discarded genuine duplicate purchase lots); only drop-Migrasi remains.
- **Reorder**: velocity = avg monthly qty over the trailing N-month window, computed as **total √∑ N months** (not √∑ the calendar buckets the date-window straddles, which is N+1 and would dilute the rate by ~1/(N+1)); 6mo window, fallback 12‚Üí24mo. Volatility CV ‚Üí safety multiplier; **lead time is per-SHOP, not per-SKU** (the forwarder ships at its own pace ‚ÄĒ AliExpress ‚Čą1mo vs the Ocistok/Martkita sea-freight forwarder ‚Čą2.5mo): each import shop's lead = `LEAD_TIME_PERCENTILE` (p75) of its observed `Tanggal Bayar`‚Üí`Tanggal Sampai` (non-Migrasi), thin shops (<`LEAD_TIME_MIN_LOTS`) ‚Üí global-import p75. **Ocistok = Martkita = 1688 are one forwarder** (`OCISTOK_KEYWORDS`/`IMPORT_SHOP_KEYWORDS`). A SKU takes the **slowest shop supplying ‚Č•`LEAD_SHOP_MIN_SHARE` of its qty** (plan for the slow import, not an occasional fast local top-up); import status comes from the reliable `Luar Negeri?`/China-keyword qty share and the forwarder from the **standardized `Toko`** value, and import SKUs are floored at the global-import lead. Local-sourced ‚Üí `LEAD_TIME_MARKET_MONTHS`. Then ROP, suggested order; action buckets STOCKOUT/URGENT/Now/Soon/Overstock + Slow/Dead. Tunables in `config.py`. See `compute_lead_time_months`.
- **Summary (00) surfacing**: the sheet flags a **partial/current year** (data not yet full-year, don't compare straight to complete years) and a **data-quality** block (OVERSOLD SKUs from the ledger + SKUs sold without HPP) so they're visible in the workbook, not just console.
- **Restock price check (`--restock-check`)**: input `data/restock_check.xlsx` (SKU, Toko, `Harga RMB` and/or `HPP IDR`, `Kompetitor Min`/`Max`). Predicts **landed** HPP from the raw RMB price via a factor **calibrated on the Ocistok/Martkita channel** (median of realized `HPP per Buah (Rp) √∑ (x RMB)` from the `Keterangan` note ‚ÄĒ per-SKU when it has ‚Č•`RESTOCK_RMB_MIN_LOTS` lots, else the global median ‚Čą Rp`RMB_TO_IDR_FALLBACK`/RMB). This factor **already embeds the forwarder's margin + shipping + import** (it runs ~25% above the raw RMB‚ÜíIDR spot `RMB_SPOT_FX_IDR`, because it's the final cost actually paid). A given `HPP IDR` overrides. Cost verdict = landed HPP vs the SKU's `hpp_wa` (¬Ī`RESTOCK_COST_TOL`). Per-marketplace selling price = `HPP √ó (1 + RESTOCK_TARGET_NET_MARKUP) / (1 ‚ąí fee)` so net profit ‚Č• target AFTER the fee; **fees derived per platform from `BisaJual`** (`|admin|/omzet`, fallback `PLATFORM_FEE_FALLBACK`). Decision vs the competitor range: ūüüĘ restock & sell (hits target within market), ūüü° thin (profitable only below target), ūüĒī don't sell (loss even at the competitor's max). See `restock_pricing.py`.
- **Cash-flow restock plan (`--cashflow`)**: turns the reorder metrics into a purchasing-budget calendar. For each SKU with demand, the **next** order is timed by when projected stock crosses `rop_final` (`months_to_order = max(0, (sisa ‚ąí rop_final) / velocity)`; STOCKOUT/below-ROP ‚Üí due now); only orders falling within `CASHFLOW_HORIZON_MONTHS` (default 6) are budgeted. Order qty reuses the reorder formula (`target_qty_post_reorder ‚ąí stock_at_order + lead_demand`, ceil); **cost = qty √ó replacement HPP** = `hpp_pricing` (latest overseas lot, the cost you'd actually pay to restock), fallback `hpp_wa`. Supplier = the SKU's dominant standardized `Toko` by non-Migrasi qty. Output `Analisa_Cashflow_Restock.xlsx`: `00_Ringkasan` (total + due-this-month + per-month), `01_Kalender_per_Bulan` (supplier √ó month Rp matrix), `02_Detail_per_SKU` (one row per order, with `Order ke-`). Zero-config (no template), so it always runs in `--all`. Plans **all** reorder cycles in the horizon via an inventory-position simulation (order up to `target + lead_demand` whenever the position crosses `rop_final`, then deplete at `velocity`), so a fast mover gets several orders, capped by `CASHFLOW_MAX_CYCLES`. See `build_restock_plan` in `cashflow.py`.
- **Per-SKU channel optimizer (`--channel`)**: for each SKU, the realized **net margin/unit** per marketplace = `(omzet + admin) / qty ‚ąí hpp_wa` (admin = `tambahan + kode_unik` from `BisaJual`, stored negative). The SKU's dominant-volume channel is compared to its best **established** channel (qty ‚Č• `CHANNEL_MIN_QTY`); a ūüĒĀ shift is flagged only when the best beats the dominant by ‚Č• `CHANNEL_SHIFT_MIN_GAP √ó hpp_wa` per pcs, with a potential-uplift estimate (`gap √ó dominant qty`). Output `Analisa_Channel_per_SKU.xlsx`: `00_Ringkasan`, `01_Rekomendasi_Channel`, `02_SKU_x_Channel`. Zero-config. See `analyze_channels` in `channel_analysis.py`.
- **Bundle / cross-sell (`--bundle`)**: market-basket on SKUs grouped by `Invoice`. For each SKU pair: `support` = orders with both, `confidence` = P(other | one) both ways, `lift` = `support √ó N_orders / (orders_A √ó orders_B)`. Pairs with support ‚Č• `BASKET_MIN_PAIR_SUPPORT` are reported (top `BASKET_TOP_N` by support then lift). Output `Analisa_Bundle_CrossSell.xlsx`: `00_Ringkasan`, `01_Pasangan_SKU`, `02_Cross_Sell_per_SKU` (per SKU's best partner by confidence). Zero-config. See `analyze_baskets` in `basket_analysis.py`.
- **Supplier classification**: China = `Luar Negeri? = 1` or the standardized `Toko` matches `CHINA_KEYWORDS` (Ocistok/Martkita, AliExpress, Jasa Impor, 1688, Alibaba, Osell); Market = `MARKET_KEYWORDS` (Shopee/Tokopedia/Bukalapak/Blibli/Tiktok ‚ÄĒ matched exactly against the clean `Toko`); else Other. The keyword lists are tuned to the standardized `Toko` values (no trailing-space hacks).
