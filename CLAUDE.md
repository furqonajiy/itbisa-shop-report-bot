# CLAUDE.md — itbisa-shop-report-bot

> **Source of truth for the Claude surfaces** — read by **Claude Code** and pasted into the **Claude Chat** project instructions (Claude has no tight char cap, so it gets the full detail). `AGENTS.md` (ChatGPT Codex) points here; `CHATGPT_CHAT.md` is the ≤8000-char condensed copy for **ChatGPT Chat** only.

Standalone, **offline** Python tool that turns ITBisa sales/stock Excel exports into an analysis workbook (sales performance, pricing, supplier, and reorder analysis + A/B price-test reports). No API calls, no network, no tokens, no GitHub Actions — runs locally and is idempotent.

> Unlike the sibling ITBisa repos, this one is **not** a GitHub-Actions bot. It does not call Shopee/TikTok Shop APIs, has no `bot-state` branch, no workflows, no Telegram, and no secrets.

## Stack & files (flat layout, no `src/`)
- Python 3.10+. Deps: `pandas`, `openpyxl` (`requirements.txt`).
- `main.py` — CLI entry point / orchestration.
- `config.py` — all constants: globs, sheet/column names, thresholds, colors, supplier keywords, reorder + A/B params.
- `data_loader.py` — multi-file glob loading, SKU normalization, current-workbook loaders (arrived beli / jual / hilang / pindah).
- `analysis.py` — HPP weighted-average, profit, per-SKU aggregation, supplier classification, reorder metrics, `build_stock_ledger`.
- `tables.py` — table builders (diminati, profit, rugi, kandidat, supplier, reorder).
- `excel_writer.py` — Excel output, incl. the per-gudang stock-reconciliation sheet.
- `ab_testing.py` — A/B price-test analyzer + template creation.
- `restock_pricing.py` — restock price evaluator (predict landed HPP, judge supplier price, recommend per-marketplace selling price) + template.
- `cashflow.py` — cash-flow restock plan: from the reorder metrics, project each order's timing/qty/cost (all cycles in the horizon) and bucket the Rupiah by month & supplier.
- `basket_analysis.py` — bundle / cross-sell market basket: SKU pairs frequently bought in one `Invoice`.
- `deadstock_analysis.py` — dead-stock / capital-release: Rupiah frozen in Overstock + Slow/Dead SKUs and how to free it.
- `momentum_analysis.py` — sales-momentum + ABC focus: accelerating vs declining SKUs, Pareto-by-profit classes.
- `elasticity_analysis.py` — price-elasticity miner: per-SKU demand sensitivity (log-log qty vs price), raise/caution recommendation.
- `trend_analysis.py` — sales trend & seasonality: cross-year omzet/profit trend, YoY growth, per-month seasonal index.
- `data/` — input Excel (gitignored). `output/` — generated reports.

## CLI (`python main.py`)
- (no flag) → **runs the full suite (same as `--all`)**: all years sales + trend + reorder + cash-flow + bundle + dead-stock + momentum + elasticity + ab-test + restock-check.
- `--sales` → all years (one file per year); `--sales YEAR` → specific year.
- `--trend` → `Analisa_Tren_Musiman.xlsx` (cross-year omzet/profit trend, YoY growth, seasonal index; zero-config).
- `--reorder` → standalone `Analisa_Reorder.xlsx`.
- `--cashflow` → `Analisa_Cashflow_Restock.xlsx` (restock budget calendar; zero-config, built from the reorder metrics).
- `--bundle` → `Analisa_Bundle_CrossSell.xlsx` (SKU pairs bought together; zero-config).
- `--deadstock` → `Analisa_Modal_Beku.xlsx` (capital frozen in slow/dead/overstock + how to free it; zero-config).
- `--momentum` → `Analisa_Momentum_ABC.xlsx` (accelerating vs declining SKUs + ABC focus; zero-config).
- `--elasticity` → `Analisa_Elastisitas_Harga.xlsx` (per-SKU price elasticity → raise/caution; zero-config).
- `--ab-test` → `Analisa_AB_Test.xlsx` (auto-creates `data/ab_tests.xlsx` template if missing).
- `--restock-check` → `Analisa_Restock_Check.xlsx` (auto-creates `data/restock_check.xlsx` template if missing).
- `--all` → all years + trend + reorder + cash-flow + bundle + dead-stock + momentum + elasticity + ab-test + restock-check (10 steps). **Loads the workbooks once (`_load_shared`) and computes reorder once**, sharing them across every step (the standalone flags still load on their own). Sales, trend, cash-flow, bundle, dead-stock, momentum, and elasticity need no template, so they always run; the ab-test and restock-check steps run **only when their template exists with rows** — otherwise they're skipped silently (no template is auto-created, so the full run never halts). Use the dedicated `--ab-test` / `--restock-check` flag to bootstrap a template.
- `--data-dir` / `--output-dir` override the defaults.

## Inputs (`data/`, matched by glob)
- `STOK_GLOB = "*BisaStok*.xlsx"` (purchases) and `JUAL_GLOB = "*BisaJual*.xlsx"` (sales). Filenames must contain `BisaStok` / `BisaJual`.
- Stok files need sheet `BisaStok`; the latest file also needs `BisaHilang` and `BisaPindahBarang` for stock reconciliation.
- Jual files need at least `BisaJualShopee`; other `BisaJual*` sheets load if present.
- BisaStok's **`Toko`** column holds the **standardized** supplier/forwarder (e.g. `Ocistok/Martkita`, `AliExpress`, `Jasa Impor Tiongkok`, `Osell`, `Shopee`, `Tokopedia`, local distributors); the payment account + invoice/resi live in **`Keterangan Pembelian`** (not read by the tool); **`Luar Negeri?`** is the authoritative overseas flag (it splits e.g. `Shopee` into local vs cross-border import). The loader auto-renames the pre-standardization header `Toko[spasi]Akun Pemesan` → `Toko` (`COL_STOK_TOKO_LEGACY`).

## Output: `output/Analisa_Penjualan_ITBisa_<year>.xlsx`
**9 sheets (pure sales history)**: `00_Summary`, `01_Paling_Diminati`, `02_Profit_Tertinggi`, `03_Barang_Rugi`, `04_Margin_Borderline`, `05_Kandidat_Naik_Harga`, `06_Per_Platform`, `07_Data_Lengkap_per_SKU`, `08_Supplier_Analysis`. **Reorder + per-gudang stock are a current-day snapshot (not year-specific), so they are NOT written into the yearly file** — they live only in `Analisa_Reorder.xlsx` (`--reorder`: `00_Reorder_Summary`, `01_Reorder_Action`, `02_Reorder_Data_Lengkap`, `03_Rekap_Stok_per_Gudang`). `ledger_df` is still passed to the yearly writer for the `00_Summary` OVERSOLD data-quality note.

## Core logic (do not regress — README has the full spec)
- **Sisa stok = current-workbook ledger**, recomputed to match the Google Sheets `BisaRekapBarang`: per (SKU, gudang) `arrived beli − nonvoid jual + ketemu − hilang ± pindah`. "Current workbook" = the latest `*BisaStok*`/`*BisaJual*` file by filename sort. Migrasi rows are the opening balance (kept here). Negative gudang balances floor to 0 and shift the deficit; truly OVERSOLD SKUs stay negative and are flagged to console.
- **HPP = weighted average with Ocistok-priority**: if a SKU has Ocistok/Martkita (China-direct) purchases, average those only; else average all. Combined across all stok files. For HPP/total-beli, Migrasi rows are dropped when real purchases exist (avoid double-count).
- **Pricing-decision basis** (markup %, Kandidat Naik Harga, Margin Borderline, Barang Rugi recommendations): `markup_pct = (harga_sekarang − hpp_pricing) / hpp_pricing`, where `hpp_pricing` = latest overseas lot price (`Luar Negeri? = 1`, by Tanggal Bayar; else `hpp_wa`) and `harga_sekarang` = lowest non-CoD unit price (Omzet/Qty) on the SKU's most recent selling day (else `harga_jual_avg`). **Profit/margin still use `hpp_wa` + `harga_jual_avg`** (realized P&L). See `_latest_ln_price` / `compute_harga_sekarang` in `analysis.py`.
- **Recent-price-increase guard (Kandidat Naik Harga)**: a SKU whose `harga_sekarang` is a freshly-raised, under-validated price is **held** — its `Harga +10/15/20%` and `Proyeksi Profit` cells are blanked and the Saran is replaced with `⏳ Harga baru naik … — kumpulkan data dulu` — because its `Qty Terjual`/`Profit` were earned at the OLD price, so extrapolating them at the new price is invalid. Change date comes from `ab_tests.xlsx` when logged (authoritative), else auto two-window step-detection; flags only when `harga_sekarang` is ≥`PRICE_CHANGE_MIN_STEP` above the pre-change price AND post-change qty is `< PRICE_CHANGE_VALIDATION_MIN_SHARE` of the year's qty. A long-stable price (e.g. Rp999 for months) is **not** flagged even if far above the lifetime average. See `compute_price_change_status` in `analysis.py`; `PRICE_CHANGE_*` tunables in `config.py`.
- **SKU normalization**: `UPPER().strip()` on load everywhere (matches case-insensitive `SUMIF`).
- **No dedup**: the old `drop_duplicates` was removed (it discarded genuine duplicate purchase lots); only drop-Migrasi remains.
- **Reorder**: velocity = avg monthly qty over the trailing N-month window, computed as **total ÷ N months** (not ÷ the calendar buckets the date-window straddles, which is N+1 and would dilute the rate by ~1/(N+1)); 6mo window, fallback 12→24mo. Volatility CV → safety multiplier; **lead time is per-SHOP, not per-SKU** (the forwarder ships at its own pace — AliExpress ≈1mo vs the Ocistok/Martkita sea-freight forwarder ≈2.5mo): each import shop's lead = `LEAD_TIME_PERCENTILE` (p75) of its observed `Tanggal Bayar`→`Tanggal Sampai` (non-Migrasi), thin shops (<`LEAD_TIME_MIN_LOTS`) → global-import p75. **Ocistok = Martkita = 1688 are one forwarder** (`OCISTOK_KEYWORDS`/`IMPORT_SHOP_KEYWORDS`). A SKU takes the **slowest shop supplying ≥`LEAD_SHOP_MIN_SHARE` of its qty** (plan for the slow import, not an occasional fast local top-up); import status comes from the reliable `Luar Negeri?`/China-keyword qty share and the forwarder from the **standardized `Toko`** value, and import SKUs are floored at the global-import lead. Local-sourced → `LEAD_TIME_MARKET_MONTHS`. Then ROP, suggested order; action buckets STOCKOUT/URGENT/Now/Soon/Overstock + Slow/Dead. Tunables in `config.py`. See `compute_lead_time_months`.
- **Summary (00) surfacing**: the sheet flags a **partial/current year** (data not yet full-year, don't compare straight to complete years) and a **data-quality** block (OVERSOLD SKUs from the ledger + SKUs sold without HPP) so they're visible in the workbook, not just console.
- **Restock price check (`--restock-check`)**: input `data/restock_check.xlsx` (SKU, Toko, `Harga RMB` and/or `HPP IDR`, `Kompetitor Min`/`Max`). Predicts **landed** HPP from the raw RMB price via a factor **calibrated on the Ocistok/Martkita channel** (median of realized `HPP per Buah (Rp) ÷ (x RMB)` from the `Keterangan` note — per-SKU when it has ≥`RESTOCK_RMB_MIN_LOTS` lots, else the global median ≈ Rp`RMB_TO_IDR_FALLBACK`/RMB). This factor **already embeds the forwarder's margin + shipping + import** (it runs ~25% above the raw RMB→IDR spot `RMB_SPOT_FX_IDR`, because it's the final cost actually paid). A given `HPP IDR` overrides. Cost verdict = landed HPP vs the SKU's `hpp_wa` (±`RESTOCK_COST_TOL`). Per-marketplace selling price = `HPP × (1 + RESTOCK_TARGET_NET_MARKUP) / (1 − fee)` so net profit ≥ target AFTER the fee; **fees derived per platform from `BisaJual`** (`|admin|/omzet`, fallback `PLATFORM_FEE_FALLBACK`). Decision vs the competitor range: 🟢 restock & sell (hits target within market), 🟡 thin (profitable only below target), 🔴 don't sell (loss even at the competitor's max). See `restock_pricing.py`.
- **Cash-flow restock plan (`--cashflow`)**: turns the reorder metrics into a purchasing-budget calendar. For each SKU with demand, the **next** order is timed by when projected stock crosses `rop_final` (`months_to_order = max(0, (sisa − rop_final) / velocity)`; STOCKOUT/below-ROP → due now); only orders falling within `CASHFLOW_HORIZON_MONTHS` (default 6) are budgeted. Order qty reuses the reorder formula (`target_qty_post_reorder − stock_at_order + lead_demand`, ceil); **cost = qty × replacement HPP** = `hpp_pricing` (latest overseas lot, the cost you'd actually pay to restock), fallback `hpp_wa`. Supplier = the SKU's dominant standardized `Toko` by non-Migrasi qty. Output `Analisa_Cashflow_Restock.xlsx`: `00_Ringkasan` (total + due-this-month + per-month), `01_Kalender_per_Bulan` (supplier × month Rp matrix), `02_Detail_per_SKU` (one row per order, with `Order ke-`). Zero-config (no template), so it always runs in `--all`. Plans **all** reorder cycles in the horizon via an inventory-position simulation (order up to `target + lead_demand` whenever the position crosses `rop_final`, then deplete at `velocity`), so a fast mover gets several orders, capped by `CASHFLOW_MAX_CYCLES`. See `build_restock_plan` in `cashflow.py`.
- **Bundle / cross-sell (`--bundle`)**: market-basket on SKUs grouped by `Invoice`. For each SKU pair: `support` = orders with both, `confidence` = P(other | one) both ways, `lift` = `support × N_orders / (orders_A × orders_B)`. Pairs with support ≥ `BASKET_MIN_PAIR_SUPPORT` are reported (top `BASKET_TOP_N` by support then lift). Output `Analisa_Bundle_CrossSell.xlsx`: `00_Ringkasan`, `01_Pasangan_SKU`, `02_Cross_Sell_per_SKU` (per SKU's best partner by confidence). Zero-config. See `analyze_baskets` in `basket_analysis.py`.
- **Dead-stock / capital release (`--deadstock`)**: quantifies capital frozen in the reorder buckets `🔵 Overstock` and `💤 Slow/Dead`. Per SKU: `held = sisa_stok × hpp_wa` (capital tied up at cost) and `freeable = max(0, sisa_stok − target_qty_post_reorder) × hpp_wa` (the actionable excess above the reorder target). Recommendation: 🧹 Likuidasi (velocity ≤ 0 or no sale in `DEADSTOCK_DEAD_DAYS`), 🏷️ Markdown (slow turnover), ⛔ Stop reorder (healthy demand but far above target). Output `Analisa_Modal_Beku.xlsx`: `00_Ringkasan`, `01_Modal_Beku_per_SKU`, `02_Per_Supplier`. Zero-config (built from the reorder metrics). See `analyze_deadstock` in `deadstock_analysis.py`.
- **Sales trend & seasonality (`--trend`)**: a cross-year view the isolated per-year files can't give. **Yearly** omzet/qty/profit/orders + YoY omzet growth (current year flagged partial); **monthly** full omzet/profit time series (gaps filled 0); **seasonality** = a per-calendar-month index where each `(year, month)` omzet is divided by that year's own monthly average (de-trends for year-over-year growth) and averaged across the **complete** years only (the partial current year is excluded so it can't bias the pattern) — index > 1 = consistently strong, classed Puncak/Normal/Sepi via `TREND_PEAK_INDEX`/`TREND_LOW_INDEX`, thin months (< `TREND_SEASONAL_MIN_YEARS`) marked. Headline also shows YTD-vs-same-period-last-year. `profit = omzet + admin − hpp_wa × qty` (no-HPP SKUs count in omzet, not profit). Output `Analisa_Tren_Musiman.xlsx`: `00_Ringkasan`, `01_Tren_Tahunan`, `02_Tren_Bulanan`, `03_Musiman`. Zero-config. See `analyze_trend` in `trend_analysis.py`.
- **Price-elasticity miner (`--elasticity`)**: per SKU, fits a log-log OLS `ln(qty_month) = a + b·ln(price_month)` over its monthly history; `b` = price elasticity. Needs ≥ `ELASTICITY_MIN_MONTHS` months and price CV ≥ `ELASTICITY_MIN_PRICE_CV` (else `⚪ Data kurang`). `|b| < ELASTICITY_INELASTIC_THRESHOLD` (inelastic) → 🔼 Naikkan harga; `|b| ≥` it (elastic) → 🔽 Sensitif harga; `b ≥ 0` → ↔ Tak konklusif. Confidence from `R²` (`Tinggi/Sedang/Rendah` via `ELASTICITY_MIN_R2`); a +`ELASTICITY_PRICE_NUDGE` price scenario models qty change `(1+nudge)^b−1` and revenue change `(1+nudge)^(1+b)−1`. **Low-confidence fits are flagged, not turned into raise recs** (observational data is confounded by season/stock/competitor). Output `Analisa_Elastisitas_Harga.xlsx`: `00_Ringkasan`, `01_Rekomendasi_Harga`, `02_Data_Bulanan`. Zero-config. See `analyze_elasticity` in `elasticity_analysis.py`.
- **Momentum + ABC focus (`--momentum`)**: two lenses to decide push vs prune. **Momentum** = qty in the last `MOMENTUM_WINDOW_DAYS` vs the prior window (🚀 Akselerasi / 📉 Menurun at ±`MOMENTUM_GROWTH_THRESHOLD`, ➡️ Stabil, 🆕 Baru naik, 💤 Berhenti; needs ≥ `MOMENTUM_MIN_QTY` total). **ABC** = Pareto by trailing profit (`omzet + admin − hpp_wa × qty` over `MOMENTUM_TRAILING_DAYS`): cumulative share ≤ `ABC_A_SHARE` = A, ≤ `ABC_B_SHARE` = B, else C. The recommendation combines them (A+🚀 → ⭐ Dorong; A+📉 → ⚠ Lindungi; C+📉 → ✂ Pangkas; …). Output `Analisa_Momentum_ABC.xlsx`: `00_Ringkasan`, `01_Fokus_SKU`, `02_ABC_Pareto`. Zero-config. See `analyze_momentum` in `momentum_analysis.py`.
- **Supplier classification**: China = `Luar Negeri? = 1` or the standardized `Toko` matches `CHINA_KEYWORDS` (Ocistok/Martkita, AliExpress, Jasa Impor, 1688, Alibaba, Osell); Market = `MARKET_KEYWORDS` (Shopee/Tokopedia/Bukalapak/Blibli/Tiktok — matched exactly against the clean `Toko`); else Other. The keyword lists are tuned to the standardized `Toko` values (no trailing-space hacks).

## Conventions
- All constants live in `config.py` — never hardcode thresholds/column names in logic.
- User-facing console strings are Bahasa Indonesia.
- **Markdown docs (`*.md`) are written in English prose**, keeping **special Indonesian terms verbatim**: Excel sheet names (`01_Paling_Diminati`, `03_Rekap_Stok_per_Gudang`, …), input sheet/column names (`BisaStok`, `Toko`, `Keterangan Pembelian`, `Luar Negeri?`, `Tanggal Bayar`, `Harga RMB`, …), domain terms (`HPP`, `Migrasi`, `omzet`), supplier/forwarder names (`Ocistok/Martkita`, `Jasa Impor Tiongkok`, …), and any Bahasa Indonesia console string quoted as an example. The console strings emitted by the code stay Bahasa Indonesia (per the bullet above).
- **The user runs Windows PowerShell** — give any CLI command/snippet you hand to the user in **PowerShell** syntax (backtick `` ` `` line-continuation, `$arrays`), not bash.
- Minimal, targeted changes only; preserve existing behavior unless explicitly in scope.

## Development workflow (process standard)
- Branch from `main` using `feature/<short-description>` (e.g. `feature/document-dev-workflow`).
- Always open a PR into `main` and **merge with a merge commit (`--no-ff`)** — never squash, never fast-forward — so the feature branch stays an ancestor of `main`.
- The merge commit title must be **representative and end with the PR number**, e.g. `Update Project Instructions to the Latest State (#47)` — never the default `Merge pull request …`.
- Commits and PRs are authored as **`C - Furqon Aji Yudhistira <furqonajiy@gmail.com>`** — at the start of every session set `git config user.name "C - Furqon Aji Yudhistira"` and `git config user.email "furqonajiy@gmail.com"` (don't wait to be asked).
- **No AI/assistant references anywhere** in branch names, commit messages, PR titles/bodies, or code/comments: no "Claude"/"Anthropic", no `Co-Authored-By` trailer, no "Generated with …" line, no model identifiers, no session links. History must read as human-authored.
- **Strip the auto-appended PR footer**: the PR-creation tool automatically appends a `_Generated by [Claude Code](…session…)_` line to the PR **body**. Immediately after opening any PR, edit the description (`update_pull_request`) to remove that footer, then re-read to confirm it's gone. Also avoid raw `<...>` placeholders in PR/issue bodies — GitHub strips them as HTML; use `[tgl]`-style instead.
- **Branch deletion isn't possible from this environment** — `git push --delete` is blocked by the sandbox git proxy (HTTP 403) and the GitHub MCP exposes no delete-ref tool. After a merge, hand the user a PowerShell `git push origin --delete <branches>` command (or the repo Branches-page link) so they can delete the merged feature branches themselves.
- Keep changes minimal and targeted; update `CLAUDE.md` / `README.md` in the same PR whenever behavior or process changes.
- **AI-instruction files**: `CLAUDE.md` (this file) = Claude Code **+ Claude Chat** · `AGENTS.md` = ChatGPT Codex (points here) · `CHATGPT_CHAT.md` = the ≤8000-char condensed brief for **ChatGPT Chat** only (ChatGPT's limit). Keep `CHATGPT_CHAT.md` in sync with this file's substance but update it **only when explicitly asked**.
- Sync marker: a file named `YYYY-MM-DD_HHMM.txt` (WIB) sits at the repo root. **On every update to this repo, rename it to the current WIB timestamp** — it signals whether the repo / Claude / ChatGPT instructions are in sync.
- Doc/marker updates (this file, `CHATGPT_CHAT.md`, the sync marker) ride in the **same feature branch and PR as the related code change** — never a separate doc-only branch (avoids noise).
