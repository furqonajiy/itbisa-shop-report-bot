# CLAUDE.md â€” laporan (Laporan generator)

> **Generator-specific deep-dive** for the `laporan/` subproject of **itbisa-shop-report-bot**. The repo-wide source of truth is the **root `CLAUDE.md`**; this file documents the generator internals. Read by **Claude Code** and **Claude Chat**.

**Offline** Python tool that turns raw marketplace exports (Shopee, Tokopedia, Tiktok, Bukalapak) into standardized **Laporan** workbooks (`Invoice` / `Jual` / `Remit` / `Bonus` + a combined `Final` sheet per period). No API calls, no network, no secrets, no GitHub Actions â€” runs locally and is idempotent. The generated **`Jual`** sheet is the upstream feed consumed by the parent bot. It is an importable **`laporan` package** inside the repo: the bot's `python main.py --laporan` calls `laporan.main.main` **in-process**, and it also runs standalone as **`python -m laporan`** (from the repo root). Every intra-package import uses the absolute `laporan.*` form, so its modules never collide with the bot's flat root modules.

## Stack & files
- Python 3.13 (runs on 3.11+). Deps: `pandas` (`>=2.0` â€” **2.x and 3.0 both supported**), `openpyxl` (`requirements.txt`).
  - **pandas 3.0 compat**: the generator's legacy readers/builders assume **object-dtype** strings (they assign ints into text columns and mutate in place), which pandas 3.0's new default `str` dtype rejects (`TypeError: Invalid value '0' for dtype 'str'`). `main()` calls `_enable_legacy_string_dtype()` â†’ `pd.set_option("future.infer_string", False)` (guarded; absent on pandas < 2.1 where object is already the default), so the generator behaves byte-for-byte identically on 2.x and 3.0 with no per-builder rewrite. Verified: full generate (all marketplaces) produces identical output on 2.3.3 and 3.0.3.
  - **2.x baseline carried forward**: the Excel append writers let pandas auto-load the workbook (no `ExcelWriter.book` setter â€” removed in 2.0); date parsing uses `to_datetime(..., format='mixed', dayfirst=...)`; forward-fills use `.ffill()`. `utility/generic.py` imports `SettingWithCopyWarning` from `pandas.errors` with fallbacks (it was removed entirely in pandas 3.0).
- `__init__.py` / `__main__.py` â€” package marker (every subpackage has an `__init__.py`) and the `python -m laporan` entry point (calls `laporan.main.main`).
- `main.py` â€” **CLI entry point + orchestration** (`laporan.main`). Parses flags (`argparse`), sets the data/report dirs, and defines `MARKETPLACE_PROCESSORS` (marketplace â†’ ordered processor modules) + `run(list_report, marketplaces=None)`. After a marketplace's processors finish, `run` calls `generate_final(<Marketplace>)` so the `Final` sheet sees every period's workbook. No `sys.path` manipulation â€” imports are absolute `laporan.*`.
- `process/preprocess.py` â€” `generate_report_list`: **recursive** glob of `data/` (`**/*.xls*`, `**/*.csv`).
- `process/<marketplace>/<vN>.py` â€” per-marketplace/version readers: drop invalid-status rows, validate keywords, dispatch to the sheet generators.
- `invoice|jual|remit|bonus|fee/` â€” each has a `generic.py` with the shared `*_to_excel` writer plus per-marketplace/version builders.
- `final/generic.py` â€” `generate_final(marketplace)`: builds the per-workbook `Final` sheet by **reading back** the already-written `Invoice`/`Jual`/`Remit` sheets (marketplace-agnostic, no per-marketplace builders).
- `rekonsiliasi/cekjual.py` â€” `reconcile_invoices(invoices, jual_dir)`: **read-only** `--cek-jual` tool. Reconciles a list of invoices (built-in default or `--invoices <file>`) against the itbisa-shop-report-bot `Jual` ledger (non-void Omzet across every `Jual*` sheet) + `Saldo` net + `Fee`, classifying Jual **entry bugs** (entry hilang / harusnya Void / salah Void / omzetâ‰ uang). Writes `reports/shopee/Cek Jual Shopee.xlsx`.
- `rekonsiliasi/generic.py` â€” `generate_reconciliation(marketplaces)`: **read-only** `--reconcile` audit. Re-reads the raw `Saldo`/`Fee` inputs, classifies every saldo row into Remit/Bonus/Penarikan/**Tidak Tercatat** using the `keywordchecker/` lists (single source of truth), and writes `reports/<marketplace>/Rekonsiliasi <Marketplace>.xlsx`. **Never changes any generated number.**
- `keywordchecker/` â€” validates marketplace status / saldo keywords (raises on unknown keyword).
- `utility/constant.py` â€” `DEFAULT_DATA_DIR` / `DEFAULT_REPORTS_DIR` (rooted at `laporan/`), `set_dirs`/`get_data_dir`/`get_reports_dir`, and `MARKETPLACE_FOLDERS` (marketplace name â†’ reports subfolder).
- `utility/generic.py` â€” `ignore_warning`, `create_directory`, `detect_marketplace`, **`build_report_path`**.
- `utility/sku.py` â€” `standardize_sku` (canonical SKU rewrites).
- `data/` â€” input exports (gitignored). `reports/` â€” generated output (gitignored).

## CLI (`python -m laporan`, from the repo root)
- (no flag) â†’ process **every** marketplace in the `MARKETPLACE_PROCESSORS` order: **Tiktok, Shopee, Tokopedia, Bukalapak**.
- `--shopee` / `--tokopedia` / `--tiktok` / `--bukalapak` â†’ process only the selected marketplace(s); flags combine.
- `--data-dir <dir>` / `--output-dir <dir>` â†’ override `./data` / `./reports`.
- `--show-files` â†’ log every discovered input file. `-v` / `--verbose` â†’ debug logging.
- `--reconcile` â†’ **read-only audit mode**: writes `Rekonsiliasi <Marketplace>.xlsx` (does **not** generate reports). Respects the marketplace + `--data-dir`/`--output-dir` flags.
- `--cek-jual` â†’ **read-only** Jual entry-bug check (`rekonsiliasi/cekjual.py`): reconciles a list of invoices against the bot's `Jual` ledger and writes `reports/shopee/Cek Jual Shopee.xlsx`; generates no reports. `--invoices <file>` supplies the invoice list (one per line, `#` comments allowed), else a built-in default.
- `--jual-dir <dir>` â†’ point `--reconcile` / `--cek-jual` at the itbisa-shop-report-bot `*Jual*.xlsx` ledger folder (for the `Cek Omzet vs Fee` sheet / the Jual check); falls back to re-derived Omzet when unset.
- With no input files found, the CLI logs `Tidak ada file ditemukan â€¦` and exits cleanly.

## Inputs (`data/`, matched recursively)
- Files may sit directly in `data/` or in any subfolder â€” discovery is recursive, and each file is classified by the **marketplace + version + type token in its filename** (not by folder), e.g. `Transaksi v2 Shopee`, `Saldo v2 Shopee`, `Fee v1 Tiktok`. Temp lock files (`~â€¦`) are skipped by the per-processor `cond` checks.
- Three input types: **`Transaksi`** (orders â†’ Invoice + Jual), **`Saldo`** (balance mutations â†’ Remit + Bonus), **`Fee`** (settlement fees â†’ feeds Shopee Remit math; is the Remit source for **Tiktok**). Legacy `Bisa*`-named raw exports (e.g. `BisaTransaksi v2 Shopee`) still classify â€” the per-processor filter matches the short type token as a substring. TikTok's `Fee` export parses in either English **or the Indonesia-localized layout** â€” `process/tiktok/v1.py` maps the Indonesia headers (`ID Pesanan/Penyesuaian`, `Total Pendapatan`, `Jumlah penyelesaian pembayaran`, â€¦) back to the canonical English names the remit math uses, and still normalizes the `Order/Adjustment ID` casing/spacing variants. TikTok's order export (`OrderSKUList`) is read **resiliently against a broken stored XLSX `<dimension>`** (some exports report only the first column; the reader re-opens the workbook non-read-only, which recomputes the real dimension, then re-reads). Shopee's `Fee` `Adjustment` detail-table header is located **dynamically** (its row floats with the preamble length) by scanning for `No. Pesanan Terhubung`, not a fixed `skiprows`.

## Output: `reports/<marketplace>/<name> Laporan <Marketplace>.xlsx`
- `<marketplace>` âˆˆ `shopee`, `tiktokshop`, `tokopedia`, `bukalapak` (lowercase; Tiktok â†’ `tiktokshop`). See `MARKETPLACE_FOLDERS`.
- The generator builds `Invoice <MP>`, `Jual <MP>` (from `Transaksi`); `Remit <MP>` (from `Saldo`, or `Fee` for Tiktok); `Bonus <MP>` (from `Saldo`); plus a `Final` sheet. Coverage: Shopee v2/v3 = all four; Tokopedia v1 = invoice + jual; Tokopedia v2 = all four; Tiktok v1 = invoice + jual + remit; Bukalapak v2 = invoice + jual + remit.
- **Deliverable sheets** (after `finalize_workbooks`): each workbook is collapsed to **`Remit <MP>`** first (the `Final` sheet promoted/renamed â€” it already combines the order + remit sides and is what gets copied into the `Jual` ledger), **`Jual <MP>`** second, and **`Bonus <MP>`** third where present. The now-redundant `Invoice <MP>` and the original `Remit <MP>` are dropped. Settlement-only workbooks (no `Final`, e.g. a period with only `Saldo`) are left untouched.

## Core logic (do not regress)
- **Path routing (`build_report_path`)**: each sheet generator still computes its report **filename** the original way â€” string-replace the input basename's type token (`Transaksi`/`Saldo`/`Fee` â†’ `Laporan`), strip ` v1/ v2/ v3` (Bukalapak keeps its ` v2`), `.csv`â†’`.xlsx`. `build_report_path` then **re-roots that filename** into `reports/<marketplace>/` (detected from the filename) and creates the folder. The directory part of the old string-replace result is discarded; only the basename is kept.
- **Workbook convergence (append model)**: for one period the `Transaksi` and `Saldo`/`Fee` files collapse to the **same** `â€¦ Laporan â€¦` basename, so all sheets land in one workbook. `Invoice` is written first in **write** mode (creates the file); `Jual`/`Remit`/`Bonus` **append** (`mode='a', if_sheet_exists='replace'`, with a create-on-missing fallback). The per-marketplace `process()` runs the `Transaksi` pass before the `Saldo`/`Fee` pass so the file exists when the appenders run. **Do not break this filename convergence or the writeâ†’append order.**
- **Recursive discovery**: `generate_report_list` returns `sorted(set(...))` of everything under `data/`; the per-processor `cond1 = '<token>' in file` filters select the right files. This is OS-independent (no Windows-only `H:\` paths or backslash globs).
- **Marketplace detection** (`detect_marketplace`): substring match of the report filename against `MARKETPLACE_FOLDERS` keys (`Shopee`/`Tiktok`/`Tokopedia`/`Bukalapak`); raises if none match.
- **Configurable dirs**: consumers call `get_data_dir()` / `get_reports_dir()` at run time (never bind the value at import) so the CLI's `set_dirs()` override (and `--data-dir`/`--output-dir`) takes effect.
- **Reconciliation (`rekonsiliasi/generic.py`, `--reconcile`)**: a **read-only audit** â€” it never edits the generated sheets. It mirrors each marketplace's `Saldo` read (skiprows/cols) and the **per-marketplace remit row filter** (Shopee v2 `#`/`Penambahan dana`, Shopee v3 `No. Pesanan` without `-`, Tokopedia `INV`, Bukalapak `#`), then buckets every row via the imported `keywordchecker/` lists: a row is **Tidak Tercatat** when it matches a remit keyword but the invoice filter excludes it (e.g. `Pencairan SPinjam untuk Penjual`), is an invoice row with no remit keyword, or is a bonus keyword where no `Bonus` is generated (Bukalapak has none). `not_used` is derived as `VALID_SALDO_KEYWORD âˆ’ components âˆ’ bonus` (stays in sync). Shopee fee check de-duplicates overlapping fee files (a yearly v2 + monthly v3 file list the same order) before matching `(Invoice, Total Penghasilan)` to the remit. Besides `Ringkasan`/`Saldo Tidak Tercatat`/`Fee Tidak Cocok`, the workbook has a **`Rincian per Deskripsi`** sheet (every row rolled up to its matched keyword â†’ category/bucket/count/total/example, uncaptured groups flagged), a **`Rincian Saldo`** row-level detail, and (Shopee) a **`Cek Remit Saldo vs Fee`** sheet listing every invoice's `Saldo` remit vs `Fee` `Total Penghasilan` side by side with a Cocok/Beda status, plus a **`Cek Omzet vs Fee`** sheet that pairs `Jual` Omzet with the real money (`Saldo` net + `Fee` `Total Penghasilan`/`Kerugian`) and flags orders whose Omzet isn't real money â€” *Retur - rugi = omzet* (return loss only in `Fee`, net 0, e.g. `231215A1CX1JNX`; especially an order **not voided** in the ledger but with no real money), *Omzet tidak settle*, *Belum ada penghasilan* (pending, not a loss). Its Omzet comes from the **itbisa-shop-report-bot** `*Jual*.xlsx` ledger (non-void `Omzet Barang`, matching the bot's Void/`Dummy` drop) when `--jual-dir` points at it (or the file sits in `data/`); otherwise Omzet is re-derived from raw `Transaksi` (status-filtered like the generator). Keep this in lockstep with the generators' filters/keywords.
- **`Final` sheet (`final/generic.py`)**: one reconciliation row per `Invoice`, joining the order side with the remit side. `Omzet Barang` = Î£ `Jual` `Omzet` per `Invoice`; `Nominal Invoice` = `Omzet Barang` + `Ongkir` + `Asuransi`. It is a **left join from `Invoice`** (every order is a row), and the remit is looked up against the **union of every `Laporan` workbook in `reports/<marketplace>/`** â€” so an order placed one period and remitted the next still matches (multi-remit â†’ amounts summed, latest `Tanggal Remit` kept); when no remit is found the remit columns stay blank. `Cek Remit`/`Untung Lainnya`/`Rugi Lainnya`/`Keterangan` are always blank (manual). Because of the cross-period lookup, `generate_final` runs **after** all of a marketplace's processors (so every period's `Remit` exists), reads the written sheets back (money is stored as text â†’ coerced), and appends/replaces the `Final` sheet (`if_sheet_exists='replace'`, create-on-missing fallback). **Do not move `Final` before the remit pass or break the cross-period union.** After all `Final` sheets are built, **`finalize_workbooks(marketplace)`** (a separate pass in `run`, *after* `generate_final`) renames `Final` â†’ `Remit <MP>` and drops the original `Invoice`/`Remit` detail. It runs last on purpose â€” the cross-period union reads the original `Remit` sheets first; and it is re-run safe because any workbook with a `Final` also has a `Transaksi`-driven `Invoice`, which the write-mode pass recreates fresh each run (so a promoted `Remit` is never mistaken for an original one).
